## Bounded Trial Configuration

The proposed trial object is the **high-resolution motorised microscope** from
the official v7.0.0-beta5 release documentation.

Proposed package ID:
`public-openflexure-microscope-v7.0.0-beta5-high-resolution`

The selection is controlled by `docs/high_res_microscope.md`, whose build
sequence identifies the high-resolution optics path, three actuators, RMS
optics, motorised operation, Raspberry Pi, Sangaboard, illumination, and
wiring steps.

In scope:

- the exact beta5 source archive recorded above;
- `CHANGELOG.md` and `okh.yml` for release/package metadata;
- `docs/high_res_microscope.md` and every build step it includes;
- `docs/buildconf.yaml` for documentation/BOM generation behavior;
- `docs/parts/mechanical.yml`, `docs/parts/electronics.yml`, and referenced
  part pages used by the selected build;
- the official pinned generated BOM at
  `https://build.openflexure.org/openflexure-microscope/v7.0.0-beta5/high_res_microscope_BOM.csv`;
- selected SCAD, STL, and DXF paths required by the configuration, by filename
  and existence only; and
- the archive license records with their separate hardware, documentation,
  and software scopes.

Out of scope:

- low-cost, upright, manual, custom, accessory, and footprint configurations;
- alternative and workaround parts unless the selected high-resolution BOM
  requires them;
- unrelated software repositories and deployment instructions;
- the unpinned `latest` URLs in `okh.yml` as controlling release sources; and
- PDF or CAD content extraction, model generation, geometry interpretation,
  or engineering validation.

## Proposed Authority Choices

| Controlled fact | Proposed authority | Allowed secondary evidence | Refusal boundary |
|---|---|---|---|
| Exact package release | Official beta5 tag and `CHANGELOG.md` beta5 section | Pinned build URL | `okh.yml` value `7.0` and any `latest` URL are less specific. |
| Selected configuration | `docs/high_res_microscope.md` in the beta5 source | Pinned generated high-resolution documentation | Other microscope configurations do not supplement this build. |
| Configuration item and quantity | Pinned `high_res_microscope_BOM.csv` | Quantity-bearing selected build steps | Part catalogs describe parts but do not control configuration quantity. |
| Part name, declared properties, and supplier reference | Referenced publisher part definition | Generated BOM label | Supplier sites do not override the pinned publisher source. |
| Controlled file existence | Preserved path plus copied-byte hash | Documentation/model reference | Existence does not establish correctness, completeness, or revision. |
| Drawing number/revision | None identified | None | Do not infer from SCAD/STL/DXF filenames or generated output. |
| Datasheet authority | None identified | Supplier and part-page links only as non-controlled references | A link is not a controlled datasheet. |

The package authority map may encode only accepted rows above. It must retain
missing authority explicitly and may not copy the pump-skid example hierarchy
by analogy.

## Canonical Identifier Rules

- Preserve all original values and YAML/CSV locators before normalization.
- Prefer a publisher part key or page anchor as the stable item identity. If a
  generated BOM row lacks one, derive an ID from category, display name, and a
  stable source-value hash; never from row order alone.
- Use one derived package-scope equipment tag, `OFM-HR-001`, for membership
  relationships. Do not present it as a publisher equipment tag or part ID.
- Build document and file-reference IDs from the in-scope relative path and a
  stable hash suffix.
- Treat `v7.0.0-beta5` as the package revision. Preserve `7.0` separately as
  less-specific Open Know-How metadata.
- Do not merge alternatives, workaround parts, tools, consumables, materials,
  or installed components merely because their names look similar.

## Expected Holds and Product Learning

An honest first working package is expected to expose
`missing_authoritative_information`, not `automatic_pass`, because drawing,
datasheet, and document-level revision authority are absent.

OpenFlexure also exposes a second product-model mismatch: quantities are
generated for a selected configuration, while part definitions live in
separate reusable catalogs. The audit must preserve both levels and the build
configuration that joined them. Treating the catalogs alone as a BOM would
lose quantity and applicability; treating the generated BOM alone as a
specification source would lose provenance.

As with the rover, an empty mandatory drawing source must not be interpreted
as proof that drawing authority is unnecessary. Working-package review must
confirm that the final state retains that missing-authority truth. This
reusable product risk is tracked as `IMP-016`.

## Preparation Acceptance Checks

The later preparation block is complete only when:

1. the source archive hash still matches the accepted intake;
2. the pinned high-resolution BOM CSV is preserved unchanged with its own
   source URL, retrieval date, size, and SHA-256;
3. the BOM is proven to belong to the exact beta5 high-resolution build;
4. every BOM row has one traceable mapping disposition;
5. every selected part definition and generated quantity retains its source
   locator and authority level;
6. alternatives and workaround parts are not silently combined with the
   selected base configuration;
7. every generated identifier and the top-level equipment tag is logged;
8. no drawing, document revision, or datasheet authority is invented;
9. no PDF/CAD content is extracted, generated, or interpreted; and
10. the prepared package is reviewed before `audit-package` is run.

## Explicit Exclusions

This definition does not authorize source conversion, running the project's
build scripts, generating models, schema or evaluator changes, a generic YAML
loader, PDF/CAD extraction, supplier-site scraping, source editing,
engineering conclusions, or audit execution.
